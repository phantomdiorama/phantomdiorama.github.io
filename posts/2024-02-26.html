My blog setup is super minimal. It’s just a single script that wraps around Pandoc. Ok there’s a second script that does some pre checks. But that’s it. I call this setup codnap. That’s Pandoc backwards. The github repo needed a name, ok?

Obviously, it’s not as automated as actual Static Site Generators like Jekyll or Hugo, but it is simpler, more flexible and allows me to keep all my blogging bits and pieces—markdown, images, css, everything—in a single folder instead of a thicket of sub-directories. This folder is called blog and is kept in Dropbox for easy backup and sync. Building the site is just a matter of running two scripts on that folder…

check

This does pre-build checks for outstanding to dos (w/ ripgrep), outstanding typos (w/ typos), markdown issue (w/ markdownlint), and displays any errors. On its face this sounds less minimal than using the eyes attached to that face, but remember behind that face is a skull of laziness. It’s fine.

It looks like this:

build

This creates a folder at blog/public if it doesn’t exist and copies all css and xml into it. Images—which I’ve pre-compressed—get copied into blog/public/images.

Then the script iterates over the list of markdown files, uses Pandoc to convert them to html, and then outputs the resulting files into blog/public too.

It looks like this:

push

After manually checking the links etc with Firefox, the contents of blog/public gets copied into a git repo which then gets pushed to the remote at my github which does the actual web hosting (for more how this part works see github pages).

wrapping up

I started off saying my setup was minimal. While it’s true it involves minimal tooling, there are a few areas which I could streamline—I hand edit the atom feed, for one. And automating image optimisation would be a nice edition. But I am loathe to change things that (mostly) work.
